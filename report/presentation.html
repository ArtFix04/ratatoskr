Ratatoskr

Onion Routing over the Yggdrasil Mesh Network

Artur Khachatryan Bachelor of Science in Computer Science American University of Armenia  ·  2025–2026
Motivation

Two Problems, One System

Problem 1 — Internet access

Yggdrasil is a private mesh — think separate internet. Nodes reach each other, but no path to the regular web. No built-in bridge. Cannot browse, resolve DNS, or reach any clearnet service.

Problem 2 — Privacy

Even with access, a single relay that sees both you and your destination breaks anonymity. Tor solves this, but its relay list is public → detectable and blockable. Central directory = single chokepoint.

Ratatoskr solves both at once: exit nodes act as internet gateways for the Yggdrasil network, and the onion circuit ensures no single node can link user to destination.

Building Block

Yggdrasil: the Transport

  • A mesh network — peers connect directly, no central server
  • Address derived from public key — address = identity, address = math
  • Address is location-independent — same address through NAT, ISP changes, travel, country moves
  • Every link already encrypted by the substrate — opaque to anyone on the wire
  • Can't fake an address — network checks the math on every link

What this gives us

Stable, unforgeable relay identifiers. No directory authority needed.

What it costs

Yggdrasil must be installed and connected to at least one peer.

Cryptography

How the Onion is Built

Three nested sealed envelopes. Built inside out — innermost for exit, then middle, then guard.

  • Each server opens only its own envelope — cannot read the others
  • Fresh lock per envelope — recording traffic now + getting key later still decrypts nothing
  • Tampered envelope is rejected outright, not silently forwarded
  • No shared key — every message is its own sealed package
# innermost: for exit
L_e = Seal(exit_pubkey,
        ONION_EXIT + dst + data)

# middle wrap
L_m = Seal(middle_pubkey,
        ONION_RELAY + addr(exit) + L_e)

# outermost: for guard
L_g = Seal(guard_pubkey,
        ONION_RELAY + addr(middle) + L_m)

# → send L_g to guard

Implementation

Technical Stack

  • Python + async IO — single loop, no threads; efficient for many concurrent relay connections
  • Encryption library = NaCl — same library used in Signal; audited C underneath, not Python
  • Web dashboard — live updates over WebSocket, no page refresh
  • Any browser works — standard proxy interface, no modifications needed
  • Gossip peer discovery — nodes ask known peers who they know; no central directory, ever

Evaluation

Does It Work?

  • test command — real circuit → fetch "what's my IP" → returned IP must be exit's, not mine
  • Demo mode — all three servers on one machine, tested with browser and curl
  • Speed: slowest link in the chain — same pattern as Tor, not a Python problem

What works

CONNECT + UDP, circuit rotation, health checking, gossip discovery, relay log, full dashboard

What does not

No traffic shaping. No hidden services. Bootstrap list is a single point of failure. One CPU core.

Context

Ratatoskr vs. Tor

Same ideas

  • 3-hop circuit, guard / middle / exit roles
  • SOCKS5 interface — existing tools work unchanged
  • Onion encryption, per-layer forward secrecy

Key differences

  • Gateway, not just privacy — Tor assumes already online; Ratatoskr gives access that wouldn't exist otherwise
  • Stronger encryption — fresh key per message vs. one key per circuit
  • Relays have no public IP — Yggdrasil handles link encryption; can't identify or block them
  • No central directory — can't block the list because there is no list

Tor is for people on the internet who want privacy. Ratatoskr is for people on the mesh who want both internet access and privacy.

What's Next

Limitations and Future Work

Discovery — most pressing

Need one known address to join. Replace with a signed, replicated peer list so anyone can bootstrap cold.

Traffic shaping

Timing correlation is currently possible. Cover traffic + fixed packet sizes would close this gap.

Host services inside mesh

Currently outbound only. Reverse circuits would let servers live inside Yggdrasil — cleaner than Tor's .onion.

Performance

Relay core in Go/Rust, same protocol. Client and dashboard stay Python. Removes the bottleneck.

Mobile

Python side is ready. Blocked on Yggdrasil mobile support, not on our code.

Formal analysis

A machine-checked proof of the SealedBox-per-layer composition. The construction is small enough to be feasible.
